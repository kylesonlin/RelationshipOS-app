# Weekly Execution Playbook
## Converting Assessments Into Immediate Action for RelationshipOS

> **🎯 Mission**: Transform weekly assessments from documentation theater into systematic execution that drives revolutionary product development.

---

## 🚀 **EXECUTION PHILOSOPHY**

### **Core Principle**
**Assessments without immediate action are expensive documentation theater.** This playbook ensures every weekly review produces specific, owned, deadline-driven tasks that advance the mission.

### **Execution Rules**
1. **Zero New Features** until all blockers resolved
2. **Specific Ownership** - no "team will handle" 
3. **Hard Deadlines** with consequences
4. **Daily Accountability** not weekly theater
5. **Evidence-Based Success** - measure resolution rates

---

## 📅 **SUNDAY NIGHT WORKFLOW**

### **Step 1: Complete Weekly Assessment**
```markdown
Run: @weekly-review

Complete the 9-section systematic assessment covering:
- Executive team analysis
- Documentation alignment audit  
- Customer reality check
- Competitive vulnerability assessment
- Technical debt & scaling audit
- Resource optimization analysis
- Assumption validation review
- Forward risk assessment
- Revolutionary standards validation
```

### **Step 2: Issue Triage**
```markdown
Run: @exec-triage

"Weekly Assessment Triage - Week [X] Issues

Take the weekly assessment report and categorize EVERY identified issue into:

🚨 BLOCKERS (Must fix this week or Week [X+1] fails):
- List specific issues that prevent next week's development
- Assign to: Reid/Jensen/Mark/Tim/Eric based on domain
- Deadline: Before any new Week [X+1] development starts

⚠️ RISKS (Must address within 2 weeks):
- List issues that could become blockers if ignored
- Assign owner and specific deadline (Week [X+1] or [X+2])
- Define success criteria for resolution

📋 IMPROVEMENTS (Address when bandwidth allows):
- List optimization opportunities
- Assign to appropriate executive for planning
- No specific deadline but track for quarterly review

For each item, provide:
□ Specific owner (Reid/Jensen/Mark/Tim/Eric)
□ Concrete deliverable 
□ Clear deadline
□ Success criteria

No vague assignments. No 'team will handle.' Every issue gets an owner."
```

### **Step 3: Ownership Clarity Review**
```markdown
Validate triage output ensures:
□ Every issue has specific named owner
□ All deadlines are concrete (not "soon" or "next week")
□ Success criteria are measurable
□ Blockers are clearly distinguished from risks
□ No issues are categorized as "note and ignore"
```

## 📊 **DAILY CHECK-IN FRAMEWORK**

### **Tuesday Check: Blocker Status**
```markdown
Question: "What blockers remain unresolved?"
Format: 
- Blocker: [Description]
- Owner: [Name]
- Status: [% complete]
- Blocker to completion: [Specific obstacle]
- Escalation needed: [Yes/No]

RULE: If any blocker <50% complete, escalate immediately.
```

### **Wednesday Check: Risk Escalation**
```markdown
Question: "Which risks are becoming blockers?"
Format:
- Risk: [Description]
- Trend: [Getting worse/stable/improving]
- Timeline to blocker: [Estimate]
- Intervention needed: [Specific action]

RULE: Any risk trending toward blocker gets immediate attention.
```

### **Thursday Check: Timeline Validation**
```markdown
Question: "Are we on track for Week [X+1] goals?"
Format:
- Goal: [Specific objective]
- Completion: [% done]
- Timeline confidence: [High/Medium/Low]
- Adjustment needed: [Scope/resource/timeline]

RULE: Low confidence triggers immediate planning adjustment.
```

### **Friday Check: Forward Blocker Prevention**
```markdown
Question: "What blockers are we creating for next week?"
Format:
- Potential blocker: [Description]
- Prevention action: [Specific task]
- Owner: [Name]
- Completion by: [Sunday night]

RULE: All potential blockers get prevention tasks assigned.
```

## 🎯 **ISSUE CATEGORIZATION FRAMEWORK**

### **🚨 BLOCKERS - Immediate Resolution Required**
**Definition**: Issues that prevent Week [X+1] development from proceeding
**Examples**:
- Database schema mismatches preventing new features
- Technical debt breaking core functionality
- Customer value gaps making demos impossible
- Resource misallocation wasting entire sprint

**Resolution Timeline**: Before any new development starts
**Escalation**: Daily until resolved

### **⚠️ RISKS - Address Within 2 Weeks**
**Definition**: Issues that will become blockers if ignored
**Examples**:
- Competitive vulnerabilities needing defensive features
- Scaling issues that break at higher load
- Assumption breaks requiring strategic pivots
- Technical architecture problems affecting future development

**Resolution Timeline**: Week [X+1] or [X+2] maximum
**Escalation**: If trending toward blocker status

### **📋 IMPROVEMENTS - Implement When Possible**
**Definition**: Optimization opportunities that enhance but don't block
**Examples**:
- Process refinements for efficiency
- Documentation gaps affecting future AI productivity
- Performance optimizations beyond requirements
- User experience enhancements

**Resolution Timeline**: When bandwidth allows
**Escalation**: Quarterly review if never implemented

## 📈 **EXECUTION METRICS & ACCOUNTABILITY**

### **Weekly Execution Metrics**
```markdown
Track these every week:
□ Blockers identified: [X]
□ Blockers resolved from last week: [X/Y]
□ Resolution rate: [%]
□ Risks elevated to blockers: [X]
□ Improvements implemented: [X]
□ Issues ignored: [X] (RED FLAG if >0)
```

### **Monthly Trend Analysis**
```markdown
Review monthly for process improvement:
□ Are we identifying problems faster than fixing them?
□ Are risks consistently becoming blockers?
□ Are improvements never getting implemented?
□ Is resolution rate improving or declining?
□ Are same issues recurring across weeks?
```

### **Success Indicators**
- Zero blockers carry forward between weeks
- Decreasing risk-to-blocker conversion rate
- Increasing improvement implementation rate
- Fewer repeat issues in assessments
- Faster resolution times week-over-week

### **Failure Indicators**
- Same issues in multiple weekly reports
- Blockers accumulating faster than resolution
- Team discussing problems without taking ownership
- Deadlines consistently missed without consequences
- Assessment becoming documentation theater

## 🚨 **ESCALATION PROTOCOLS**

### **When To Escalate**
- Blocker resolution <50% complete by Wednesday
- Risk trending toward blocker status
- Timeline confidence drops to "Low"
- Same issue appears in 2+ weekly reports
- Owner missing deadlines without communication

### **Escalation Paths**
- **Technical Issues**: @jensen → @exec if architecture decision needed
- **Strategic Issues**: @reid → @exec if market pivot required  
- **Business Issues**: @mark → @exec if model change needed
- **Operational Issues**: @tim → @exec if process change required
- **Coordination Issues**: @eric → @exec for team alignment

### **Escalation Actions**
- Immediate owner reassignment if needed
- Resource reallocation to critical blockers
- Scope reduction to ensure blocker resolution
- Timeline adjustment with strategic approval
- Process improvement to prevent recurrence
